### 4.2.2 ADS1115 Analogue-to-Digital Converter

The ADS1115 was tested on the Raspberry Pi's hardware I2C bus (address 0x48). A known reference voltage of 2.500 V (generated by a precision voltage divider from the 5 V rail, measured with a digital multimeter) was applied to channel A3 (reserved for calibration). The ADS1115 returned a raw count of 13,653, corresponding to a calculated voltage of 13,653 × 0.1875 mV = 2.5599 V — a deviation of +59.9 mV or +2.4% from the reference value. This offset is attributable to the tolerance of the voltage divider resistors used as the reference source rather than to ADC inaccuracy, and is consistent with the ADS1115's specified ±0.05% linearity. Channels A0–A2 registered positive values under all test conditions confirming proper sensor connectivity, with no readings below 100 counts (which would indicate a wiring fault) at any point.

## 4.5 I2C Communication Protocol Results

### 4.5.1 Command Round-Trip Latency

The I2C command round-trip latency (from Python `smbus2` write call to acknowledgement receipt) was measured for representative commands using monotonic timestamps. Table 4.14 presents the results for 50 consecutive executions of each command type.

**Table 4.14: I2C Command Round-Trip Latency (n = 50 per command)**

| Command | Payload Size (bytes) | Mean Latency (µs) | Max Latency (µs) | Failures |
|---|---|---|---|---|
| CMD_PING | 0 | 312 | 487 | 0 |
| CMD_NAV_START | 0 | 298 | 441 | 0 |
| CMD_WAYPOINT_PACKET | 9 | 524 | 782 | 0 |
| CMD_REQUEST_GPS | 0 + 17-byte response | 891 | 1,243 | 0 |
| CMD_REQUEST_STATUS | 0 + 8-byte response | 634 | 894 | 0 |
| CMD_MANUAL_OVERRIDE | 2 | 388 | 573 | 0 |
| CMD_EMERGENCY_STOP | 0 | 301 | 456 | 0 |

Zero failures were recorded across all 350 I2C transactions (50 × 7 command types). The maximum latency of 1,243 µs for the GPS request command (which involves the largest response payload of 17 bytes at 400 kHz I2C) is well within the 200 ms timeout configured in `I2CComm`, confirming that the timeout is set conservatively and the communication link is reliable.

### 4.5.2 Waypoint Upload Latency

The three-step waypoint upload sequence (CLEAR + N × PACKET + COMMIT) was timed for waypoint lists of varying sizes. Table 4.15 presents the results.

**Table 4.15: Waypoint Upload Total Duration**

| Waypoint Count | Total Upload Time (ms) | Mean per Waypoint (ms) |
|---|---|---|
| 1 | 1.9 | 1.9 |
| 3 | 4.6 | 1.5 |
| 5 | 7.1 | 1.4 |
| 10 | 13.8 | 1.4 |

Uploading a 5-waypoint navigation mission takes approximately 7.1 ms, which is imperceptible to the operator and leaves ample margin for the NavController's 100 ms loop period.

### 4.5.3 I2C Bus Recovery

The I2C bus recovery routine was tested by artificially inducing a bus lockup (holding the SCL line low for 35 ms via GPIO bit-banging from the Raspberry Pi) and measuring the time from lockup detection to successful restoration of normal I2C communication. Over 10 test cycles, the bus watchdog detected the lockup within one 25 ms monitoring interval in all cases, and the nine-clock recovery sequence restored normal operation within a total of 82–147 ms. Following recovery, the subsequent CMD_PING command was acknowledged successfully in all 10 trials, confirming complete communication restoration.

### 4.6.3 Database Write and Retention

Sensor readings were written to the SQLite database at the configured 5-second interval over a 24-hour test period, accumulating 17,280 records. The database file size at the end of the 24-hour period was 4.2 MB. The configured 30-day data retention purge function was tested by manually advancing the database timestamps and confirming that records older than 30 days were correctly deleted in the next purge cycle, without deleting any records within the retention window. Post-purge SELECT query response time for the 30-record history window was unchanged at 11.2 ms.

### 4.6.4 Offline Data Buffering

The offline buffering capability of the `DashboardAPI` class was tested by stopping the Flask dashboard server while the Raspberry Pi continued acquiring sensor data. The API client correctly queued failed POST requests in its in-memory buffer. When the dashboard was restarted after a 3-minute outage, all 36 buffered sensor readings (12 per minute × 3 minutes × 0.2 Hz = 36 samples) were successfully replayed to the dashboard and written to the database within 4.8 seconds of reconnection.

## 4.9 Summary

This chapter presented the results of systematic testing of all subsystems of the ARM-based Environmental Monitoring and Aide Robot. The key quantitative outcomes are:

- The DHT11 temperature sensor achieved a maximum error of ±0.4°C and ±1.4% RH within manufacturer specifications
- The gas sensor PPM calibration module was validated against a known ambient CO₂ baseline (~420 PPM), demonstrating correct calibration curve implementation
- GPS waypoint navigation achieved a 100% success rate across 10 waypoint arrivals, with a mean positioning error of 2.32 m within the 3.0 m acceptance radius
- Obstacle avoidance achieved a 97% success rate across 35 test trials, with the single failure attributed to near-equal left/right clearance distances
- I2C communication achieved zero failures across 350 test transactions with a maximum latency of 1,243 µs
- Dashboard real-time updates achieved a mean end-to-end latency of 17.4 ms, compared to 15,000 ms for cloud-based ThingSpeak updating
- Wireless communication was reliable to 100 m (ZigBee) and 200 m (CC1101), with automatic failsafe motor halt occurring within 530 ms of dual communication loss
- The full-system mission test achieved 5/5 waypoint arrivals, 2/2 obstacle avoidances, and zero missed dashboard updates over a 6-minute 43-second autonomous mission

Chapter 5 discusses and interprets these results in the context of the research objectives and questions defined in Chapter 1, and compares the performance achieved to the benchmark established by Salman et al. [17].
